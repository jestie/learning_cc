// 001 — session format
The 30-minute agenda
Every session follows this structure. No exceptions — consistency is what compounds.
0:003 min
Quick frame
Alignment
Presenter states the problem they solved, which architecture layer it lives in, and the one thing they want everyone to take away. No context-setting, no backstory — just the frame.
3:009 min
Show the build
Share
Live demo or code walkthrough. Follow the project template format so everyone knows where to look. Focus on the interesting decisions — skip the boilerplate. The repo link goes in the chat before you start.
12:006 min
The key insight
Build
One diagram, one slide, or one whiteboard moment. What's the reusable pattern? What was the hardest thing to figure out? If you can't say it in 60 seconds, distil further.
18:007 min
Open questions & riffs
Share
Group asks: "What if we extended this to…?" or "I had a similar problem…". Ideas are captured but not rabbit-holed. Presenter keeps a parking lot — anything unresolved goes there, not into the next agenda slot.
25:005 min
Commit + next step
Reflect
Each person says one thing they'll try or build before next session. Presenter posts code and README to the shared repo before EOD. This 5 minutes is what makes the other 25 compound.
// 002 — shared mental model
AI architecture layers
Every project maps to a layer. This is the shared vocabulary — use it when framing your session.
01
Foundation — models & APIs
the brain
⌄
Raw language models, embeddings, and multimodal capabilities. You don't build this — you call it. Understanding context windows, token costs, temperature, and model selection lives here. Start every project by asking: which model is right for this task, and why?
Claude / GPT / Gemini Embeddings Fine-tuning Multimodal Context windows
02
Orchestration — prompts & chains
the craft
⌄
How you structure calls to the model. System prompts, few-shot examples, chain-of-thought, structured output parsing. The difference between a bad and great AI product often lives entirely in this layer. Prompt engineering is real engineering.
LangChain / LlamaIndex Prompt templates JSON output RAG pipelines Few-shot examples
03
Memory & context — storing state
the recall
⌄
Models are stateless — you add memory. Vector databases, document stores, conversation history management, and retrieval strategies. How you feed the right context at the right moment determines everything. Retrieval quality beats model quality, often.
Pinecone / Chroma / pgvector Chunking strategy Hybrid search Reranking
04
Agents & tools — taking action
the hands
⌄
The model decides to call external tools — APIs, code runners, browsers, databases. Agentic loops, tool use, function calling, MCP servers. This is where AI goes from answering to doing. The hard problem is reliability in multi-step chains.
Tool / function calling MCP servers Code execution Browser agents ReAct loops
05
Interface — how humans interact
the surface
⌄
The surface your users touch. Chat UIs, voice, CLI tools, background automation, Slack bots. Streaming responses, feedback loops, and UX patterns specific to AI interactions. Good interfaces make the model feel smarter than it is.
Streaming UI Voice / speech CLI tools Slack / email bots Dashboards
06
Evaluation — knowing if it works
the mirror
⌄
How do you know your AI is actually good? Evals, benchmarks, logging, human feedback, A/B testing prompts. This layer is skipped most often and regretted most often. Every production AI system needs this. Build it early.
LLM-as-judge Golden datasets Logging / traces Human review A/B prompts
// 003 — project template
README format
Paste this into your repo before each session. Consistency here means everyone can pick up anyone's work cold.
README.md — standard fields
project_name
"Email summariser with memory" — one clear noun phrase
architecture_layer
Which layer(s) from the shared model does this touch? Be specific.
the_problem
One sentence. What real friction does this remove? Who feels that friction?
how_it_works
3–5 bullets. No jargon. What would you tell a friend at the pub?
the_key_decision
What was the hardest design choice? What did you try first that didn't work?
to_run_it
3 commands max. If it needs more, automate the rest. Include env var setup.
build_on_this
2–3 explicit ideas for where the group could take it next. Be concrete.
would_do_differently
Honest reflection. This field is what turns a project into a learning resource.
// 004 — working agreements
Group principles
How the group operates. Agree on these together, then hold each other to them.
// 01
Build on the same stack
Everyone anchors to the shared architecture layers. Even small experiments should map to a layer, so knowledge compounds across sessions — not just within them.
// 02
One level harder each time
Each session should push one layer deeper or wider than the last. No covering the same territory without a meaningfully new angle. Progress is the metric.
// 03
Code is the artefact
Every session produces something runnable. Slides alone don't count. The repo is the product. If it's not in the repo by EOD, it didn't happen.
// 04
Questions over polish
Half-built with honest questions beats finished with no curiosity. The best sessions end with more questions than they started. Ship rough, think deep.
// 05
Rotate the presenter
Everyone presents, everyone builds. Watching is learning; doing is knowing. Rotate weekly — no one sits in the audience for more than two sessions straight.
// 06
Document to teach
Your README should let any member run the project cold, six months from now. Write for Future-You who remembers nothing about this session.
